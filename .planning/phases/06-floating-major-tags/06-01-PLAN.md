```yaml
name: release-please

on:
  push:
    branches: [main]

permissions:
  contents: write
  pull-requests: write

concurrency:
  group: release-please-${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: false

jobs:
  release-please:
    runs-on: ubuntu-latest
    outputs:
      releases_created: ${{ steps.release.outputs.releases_created }}
      paths_released: ${{ steps.release.outputs.paths_released }}
    steps:
      - uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd # v6.0.2
        with:
          fetch-depth: 0

      - uses: googleapis/release-please-action@45996ed1f6d02564a971a2fa1b5860e934307cf7 # v5.0.0
        id: release
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          config-file: release-please-config.json
          manifest-file: .release-please-manifest.json

  tag-floating-major:
    needs: [release-please]
    if: needs.release-please.outputs.releases_created == 'true'
    runs-on: ubuntu-latest
    strategy:
      matrix:
        component: ${{ fromJSON(needs.release-please.outputs.paths_released) }}
    steps:
      - uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd # v6.0.2
        with:
          fetch-depth: 0

      - name: Force-move floating major tag
        shell: bash
        run: |
          COMPONENT="${{ matrix.component }}"
          REGION="$(basename "$COMPONENT")"
          NEW_VERSION="$(jq -r ".[\"$COMPONENT\"]" .release-please-manifest.json)"
          MAJOR="${NEW_VERSION%%.*}"
          FLOATING_TAG="${REGION}-v${MAJOR}"

          git tag -fa "$FLOATING_TAG" -m "Floating major tag for ${REGION} v${NEW_VERSION}"
          git push origin --force "$FLOATING_TAG"
          echo "::notice::Moved ${FLOATING_TAG} -> ${REGION} v${NEW_VERSION}"
```

**Implementation notes (DO NOT deviate):**

- **`needs: [release-please]`** — Strict ordering: runs only after release-please completes. Per D-11 from Phase 5 CONTEXT.md ("no placeholder job needed — Phase 6 adds second job with needs dependency")
- **`if: needs.release-please.outputs.releases_created == 'true'`** — Fast-skip: if no component released, the matrix is empty and every job is skipped. But this guard also provides a clear signal in the Actions UI that the job intentionally did not run.
- **`matrix.component: ${{ fromJSON(needs.release-please.outputs.paths_released) }}`** — Dynamic matrix: only iterates over components that actually released. If one region releases, one matrix job runs. If none, zero jobs run (matrix is empty).
- **`fetch-depth: 0`** — Same as the release-please job. Required so the target commit (referenced by the new release tag) is available locally for `git tag -fa`.
- **`basename "$COMPONENT"`** — Converts `actions/nordvpn-es` → `nordvpn-es`. This is the only safe way to extract the component name — avoids fragile string matching on `paths_released` (per REL-07 requirement "never substring-matches paths_released").
- **`jq -r ".[\"$COMPONENT\"]" .release-please-manifest.json`** — Reads the post-release version from the manifest. This file is committed to `main` by the merged release PR, so it's always the canonical source. Per CLAUDE.md §Stack: more robust than reading per-component `--tag_name` outputs across job boundaries.
- **`NEW_VERSION%%.*`** — Shell parameter expansion: extracts major from `1.2.3` → `1`. Works with pre-major too — `0.1.0` → `0`. No `jq` or `cut` needed.
- **`git tag -fa "${FLOATING_TAG}"`** — Force-creates (or force-moves if exists) the local floating tag. `-f` overwrites an existing floating tag at the old SHA.
- **`git push origin --force "${FLOATING_TAG}"`** — Force-pushes the floating tag to the remote. `--force` is required because the tag might already exist at a different SHA from a previous release.
- **No bare `v<MAJOR>` tag** — The floating tag is always `nordvpn-<region>-v<MAJOR>` (e.g., `nordvpn-es-v1`). Never `v1` alone. Per REL-08 and CLAUDE.md §"What NOT to Use": "Only per-region major floats."
- **`shell: bash`** — Explicit on the step so the runner uses bash even if the default shell is ever changed at the workflow level.
- **`::notice::`** — Prints a visible message in the Actions UI so maintainers can see which tags were moved.
- **No permissions override** — `permissions: contents: write` is already declared at the workflow level (inherited by `tag-floating-major`). `GITHUB_TOKEN` is sufficient for force-pushing tags — no PAT needed (per CLAUDE.md §Stack line 71).
- **The `release-please` job is EXACTLY as-is from Phase 5** — do not modify any existing line. Only append the new job block. No changes to triggers, permissions, concurrency, or the release-please job.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| release-please job → tag-floating-major job | Cross-job output dependency via `needs: [release-please]` and GitHub Actions output context |
| tag-floating-major job → git origin | Force-push of floating major tags to remote via `GITHUB_TOKEN` with `contents: write` |
| `.release-please-manifest.json` → tag-floating-major step | Version source of truth read at runtime via `jq` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Spoofing (S) | tag-floating-major job trigger | mitigate | Job runs only via `needs: [release-please]` — cannot trigger independently. Workflow only triggers on push to `main` with branch protection requiring `actions-lint` + `self-test`. Malicious pushes must pass all checks first. |
| T-06-02 | Tampering (T) | Floating tag value derivation | mitigate | Floating tag computation reads `.release-please-manifest.json` via `jq` and extracts major via shell parameter expansion — no dynamic interpolation of untrusted input. The manifest is git-committed, not attacker-writable. |
| T-06-03 | Tampering (T) | `release-please.yml` modification | mitigate | Changes to workflow file require PR + branch protection. The `tag-floating-major` job only runs code checked out from the repo — never from an external source. |
| T-06-04 | Repudiation (R) | Tag creation | accept | All `git push` operations are attributable: `GITHUB_TOKEN` maps to `github-actions[bot]`, and GitHub's audit log records every tag creation. Push event appears in the repo's Activity feed. |
| T-06-05 | Elevation of Privilege (E) | GITHUB_TOKEN with `contents: write` mutating tags | accept | Required contract — release-please already has `contents: write` for creating release tags. The same scope is needed for floating major tags. Token is repository-scoped and expires after workflow run. No path to cross-repo privilege escalation (same threat surface as T-05-06 from Phase 5). |
| T-06-06 | Denial of Service (D) | Force-push overwriting a floating tag mid-consumer-run | accept | Floating major tags are inherently mutable — this is their documented contract (PROJECT.md §Out of Scope: "Bare v1/v2 floating tags: Unambiguous only in a single-action repo"). Consumers are warned in every README that SHA pinning is recommended for release-critical workflows. A force-push during an active consumer workflow run is theoretically possible but practically negligible — GH Actions resolves `uses:` refs at workflow start, not at each step. |
</threat_model>
