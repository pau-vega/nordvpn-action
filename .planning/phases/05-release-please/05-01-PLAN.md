Create `.github/workflows/release-please.yml` with EXACTLY this content:

```yaml
name: release-please

on:
  push:
    branches: [main]

permissions:
  contents: write
  pull-requests: write

concurrency:
  group: release-please-${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: false

jobs:
  release-please:
    runs-on: ubuntu-latest
    outputs:
      releases_created: ${{ steps.release.outputs.releases_created }}
      paths_released: ${{ steps.release.outputs.paths_released }}
    steps:
      - uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd # v6.0.2
        with:
          fetch-depth: 0

      - uses: googleapis/release-please-action@45996ed1f6d02564a971a2fa1b5860e934307cf7 # v5.0.0
        id: release
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          config-file: release-please-config.json
          manifest-file: .release-please-manifest.json
```

**Why this exact shape:**

- **Trigger (`on.push.branches: [main]`):** Run on every push to main. No `paths:` filter — let Conventional Commit scopes route to the correct package (per user discretion in CONTEXT.md §Agent's Discretion, and the standard release-please monorepo pattern). No `pull_request` trigger — release-please generates release PRs, it doesn't react to them.

- **Permissions elevated to `contents: write, pull-requests: write`:** Required by REL-04. The action needs `contents: write` to create tags and `pull-requests: write` to open/update release PRs. This is escalated at workflow level (not per-job) since there's only one job — if the second job in Phase 6 needs different permissions, it will set `permissions` on its own job stanza per the established pattern in self-test.yml (`drift-issue` job has `permissions: issues: write`).

- **Concurrency `cancel-in-progress: false`:** Prevents simultaneous release-please runs from racing on the same ref. `cancel-in-progress: false` so a running release-please isn't killed mid-execution (consistent with self-test.yml pattern — release-please operates on main branch state, killing it mid-run could leave partial state).

- **Single job with `outputs`:** Outputs `releases_created` (boolean) and `paths_released` (JSON array string). These are consumed by Phase 6's `update-major-tags` job via `needs: release-please`. The outputs are declared here so Phase 6 can wire in without modifying this job.

- **Checkout with `fetch-depth: 0` (per REL-05):** The release-please-action reads `config-file` and `manifest-file` from the workspace — without checkout, these files don't exist on disk. `fetch-depth: 0` fetches full history, which release-please uses to compute changelogs from commit history. Even though the action primarily uses the GitHub API, having the full git history available is defensive and explicitly required by REL-05. The `de0fac2e` SHA is the canonical checkout v6.0.2 pin from CLAUDE.md.

- **Pinned SHA:** `googleapis/release-please-action@45996ed1f6d02564a971a2fa1b5860e934307cf7 # v5.0.0` — canonical v5.0.0 SHA from CLAUDE.md. NOT the archived `google-github-actions/release-please-action` (PITFALL §7).

- **No `pull_request_target`:** Single `push` trigger — no PR event at all, so no fork-safety concern. Consistent with the repo-wide ban on `pull_request_target` (PITFALL §2).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub → GitHub Actions Runner | Push to `main` triggers `release-please.yml` execution |
| GitHub Actions Runner → GitHub API | `GITHUB_TOKEN` used to create release PRs, tags, and releases |
| `.github/release-please-config.json` → release-please-action | Config controls tag shape, changelog sections, bootstrap; read at runtime |
| `.release-please-manifest.json` → release-please-action | Manifest controls version state; read/written by release-please |
| `main` branch → release-please | Conventional Commit history on main drives release decisions |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
